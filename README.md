# Personal Website API

TypeScript + Express + PostgreSQL API for the personal website maintenance module.

## Install

```bash
npm install
```

This archive intentionally omits `node_modules` and `.env` so you can generate a clean local install on your machine.

## Setup

Copy one of the example env files to `.env` and update the values.

### Local development

```bash
cp .env.development.example .env
```

### Production / VPS

```bash
cp .env.production.example .env
```

You can also start from the generic template:

```bash
cp .env.example .env
```

Run the database migrations before starting the API.

```bash
npm run migrate
```

The server also checks for and applies pending migrations on startup, which is convenient for local development and small single-instance deployments.

## Scripts

```bash
npm run migrate         # apply pending database migrations
npm run migrate:status  # show applied vs pending migrations
npm run migrate:create -- add_new_feature_table
npm run dev             # start in watch mode with tsx
npm run build           # compile TypeScript to dist/
npm start               # run compiled output
npm test                # run unit + integration tests
npm run typecheck       # run TypeScript without emitting files
```

## Environment variables

- `NODE_ENV` - `development` or `production`
- `DATABASE_URL` - PostgreSQL connection string
- `ADMIN_BOOTSTRAP_USERNAME` - initial admin username
- `ADMIN_BOOTSTRAP_PASSWORD` - initial admin password for first boot only
- `AUTH_SESSION_DAYS` - session lifetime in days
- `AUTH_RATE_LIMIT_WINDOW_MS` - login rate-limit window in milliseconds
- `AUTH_RATE_LIMIT_MAX` - maximum failed login attempts per IP inside the window
- `CORS_ALLOWED_ORIGINS` - comma-separated frontend origins or `*`
- `REQUEST_BODY_LIMIT` - Express request body size limit
- `TRUST_PROXY` - set to `true` when running behind a reverse proxy such as Nginx
- `SESSION_PRUNE_INTERVAL_MINUTES` - expired session cleanup interval
- `GRACEFUL_SHUTDOWN_TIMEOUT_MS` - force-exit timeout during shutdown
- `LOG_LEVEL` - `debug`, `info`, `warn`, `error`, or `silent`

## Local development CORS

For local Vue development, keep this in `.env`:

```env
CORS_ALLOWED_ORIGINS=http://localhost:3000,http://127.0.0.1:3000,http://localhost:5173,http://127.0.0.1:5173
```

This works together with your Vite proxy setup.

## Production / Hostinger VPS notes

If your SPA is deployed to:

- `https://domtarang.com`
- `https://www.domtarang.com`

and your API is deployed to:

- `https://api.domtarang.com`

then your production `.env` should include:

```env
NODE_ENV=production
CORS_ALLOWED_ORIGINS=https://domtarang.com,https://www.domtarang.com
TRUST_PROXY=true
```

The API now fails fast in production if `CORS_ALLOWED_ORIGINS` is missing, so deployment errors are clearer.

### Nginx

A sample Nginx config is included at:

- `deploy/nginx/api.domtarang.com.conf`

It proxies `api.domtarang.com` to this Node app on port `3000`.

### PM2

A sample PM2 config is included at:

- `ecosystem.config.cjs`

Typical production flow:

```bash
npm install
npm run build
pm2 start ecosystem.config.cjs
pm2 save
```

## Project structure

- `migrations/` - ordered SQL migrations tracked in the database
- `src/server.ts` - bootstraps the server, runs migrations, and handles graceful shutdown
- `src/app.ts` - Express app setup and middleware registration
- `src/config/` - environment and database config
- `src/routes/` - route definitions
- `src/controllers/` - HTTP handlers
- `src/services/` - business logic
- `src/repositories/` - PostgreSQL queries
- `src/middleware/` - Express middleware
- `src/validators/` - request validation logic
- `src/types/` - domain and infrastructure type definitions
- `src/utils/` - shared helpers
- `test/` - unit and integration tests executed through `tsx`
- `dist/` - compiled JavaScript output generated by `npm run build`
- `deploy/nginx/` - sample reverse proxy config for deployment

## Highlights

- strict TypeScript configuration with modern `NodeNext` module resolution
- layered Express architecture: routes, controllers, services, repositories
- centralized error handling and request validation
- Helmet-based security headers and origin-aware CORS
- login rate limiting for brute-force protection
- structured application and request logging
- forward-only SQL migrations with checksum validation
- graceful shutdown and session maintenance
- route-level integration tests with Supertest
